# 1. Purpose

This file formalizes the reusable close protocol standard for HAMAL / AMA Mob client operating environments.

The close protocol turns client clarity into a clean, scoped, professional build activation path.

Standard close flow:

```text
Lead / warm prospect
  -> Clarity intake
  -> Blueprint extraction
  -> Blueprint review / diagnosis reveal
  -> Offer summary or proposal
  -> Updated SOW / detailed scope
  -> Signature / acceptance
  -> Payment milestone 01
  -> Build activation
```

---

# 2. Close Protocol Doctrine

The close is not just a payment request.

It is where:
- the client feels understood
- the system gap becomes obvious
- deliverables become tangible
- scope becomes protected
- payment terms become clear
- support and handover expectations are defined
- reusable IP boundaries are protected
- build activation becomes legitimate

# 5. Pricing + Value Discipline

Internal value standard:

```text
The system architecture must be premium. The scope must remain disciplined.
```

Do:
- frame the work as system installation or client-flow foundation
- protect the premium nature of the architecture
- distinguish first build layer from full operating system
- track strategic, build, design, documentation, and coordination effort
- maintain costing and margin discipline
- make discounts or alignment structures strategic, not casual

Do not:
- undersell the build as a simple funnel
- overpromise a full OS when scope is a first build layer
- imply unlimited revisions
- include open-ended strategy access
- include every possible feature because it can be built
- treat performance alignment as a replacement for fair build fees

---

# 6. Payment Structure Standard

Payment structures must be explicit.

Possible structures:
- paid in full
- deposit plus balance
- milestone schedule
- pilot terms
- retainer
- approved alignment / revenue share
- pending review

Payment structure should define:
- total build fee or approved pricing posture
- milestone amounts
- milestone triggers
- when work begins
- what pauses the timeline
- what happens if payment is delayed

Do not add live payment links or processor configuration here.

---

# 7. Support Terms Standard

Support terms should define:
- post-launch support window
- support channel
- response expectations
- included meetings or async reviews
- what is excluded
- what becomes new scope
- handoff support expectations

Avoid open-ended support language.

---

# 8. Handover Rules Standard

Handover must define:
- final deliverables
- admin access transfer where applicable
- asset exports
- platform list
- SOP or walkthrough
- ownership transfer where applicable
- internal-only items not included
- client acceptance rule

Core handover rule:

```text
Client-specific outputs = client-owned.
Reusable architecture = ours.
```

---

# 9. Revision Rules Standard

Revision rules must define:
- number of included revision rounds
- revision window
- what counts as a revision
- what counts as new scope
- approval process
- impact on timeline

Do not leave revisions unlimited or undefined.

---

# 10. Risk Level Standard

Suggested values:
- low
- medium
- high
- legal_review
- financial_review
- founder_review

Use higher risk when:
- terms are unclear
- support is open-ended
- revenue share is involved
- performance language could sound like a guarantee
- ownership/IP boundaries are unclear
- client assets or access are missing
- delivery capacity is uncertain

---

# 11. Performance Framework Standard

Performance language should usually be framed as:
- benchmark
- target
- review framework
- measurement plan
- attribution method

Do not promise guaranteed outcomes unless the system controls distribution, inputs, execution, and measurement.

Revenue alignment must be:
- attributable
- time-bound
- clearly defined
- approved before sending

---

# 12. Build Activation Trigger

Build activates only after:
- SOW / acceptance or approved equivalent
- milestone payment received or explicitly waived
- required access and assets provided or logged as pending
- client registry updated
- build status marked active
- source pack saved
- lead agent has current handoff
