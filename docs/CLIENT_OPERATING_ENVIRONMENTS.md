# 1. Purpose

This file formalizes the reusable client operating environment doctrine for HAMAL / AMA Mob.

A client operating environment is not just a client folder, Notion page, Airtable record, or repo. It is the structured container where a client relationship becomes:
- source-grounded
- scoped
- priced
- documented
- built
- updated
- handed off
- reusable where appropriate

The operating truth:

```text
We are not delivering random consulting.
We are not selling websites.
```

# 4. Environment Layers

## 4.1 GitHub = Doctrine Of Truth

GitHub stores:
- project doctrine
- source indexes
- SOW versions
- architecture docs
- schemas
- specs
- prompt libraries
- playbooks
- decision logs
- handoff docs

Rule:

```text
If a doctrine, process, architecture decision, or handoff matters long-term, it must land in GitHub.
```

## 4.2 Airtable = Operational Backend

Airtable is the structured operational state layer.

Use Airtable for records, status, views, filtering, reports, and future automations only when live-system changes are approved.

This file does not create or modify Airtable.

## 4.3 Notion = Human Dashboard / Client-Facing Layer

Notion is the readable workspace layer for client dashboards, notes, roadmaps, and context pages.

This file does not create or modify Notion.

## 4.4 Graphify = Persistent Context Layer

Graphify maps relationships across docs, repos, assets, SOWs, source notes, and handoffs.

Use it before major structural decisions when a graph exists. Do not pretend a graph exists when it does not.

## 4.5 NotebookLM = Client Update + Narrative Briefing Layer

NotebookLM supports:
- weekly or milestone briefings
- client update podcasts
- internal alignment explainers
- source-grounded summaries

NotebookLM outputs must point back to source files and approved client context.

# 6. Build Activation Gate

Build begins only after:
- SOW / acceptance or approved equivalent exists
- first milestone payment is received or explicitly waived
- required access and assets are provided or logged as pending
- client record is marked build active
- client repo or project folder exists
- client workspace exists or is planned
- project source pack is saved
- lead agent / Command Center has current handoff

No build should start from chat memory alone.

---

# 7. Client Update Rhythm

Each active client environment should define:
- update cadence
- update owner
- what changed since last update
- what is complete
- what is blocked
- what is needed from client
- what happens next
- whether NotebookLM briefing is useful

Suggested update structure:
1. What this project is ultimately building
2. What has been completed
3. Why those decisions matter
4. What changed or was clarified
5. What is being built next
6. What is needed from the client
7. What to expect before the next update
